title: Pre-launch checklist
description: Environment, billing, email, database, and security items before you ship the SaaS starter to production.
---

Use this list with `.env.example` and your hosting provider‚Äôs dashboard. The codebase does not ship a production Dockerfile; deployment steps are outlined in [Deployment](./deployment.mdx).

## Environment and URLs

- **`BASE_URL`** ‚Äî Set to your public HTTPS origin (no trailing path), e.g. `https://app.example.com`. Used for auth callbacks and absolute links.
- **`POSTGRES_URL`** ‚Äî Production PostgreSQL connection string. Run migrations against this database before traffic hits the app (`pnpm db:migrate` or your CI job).
- **`AUTH_SECRET`** ‚Äî Replace any development value with a long, random secret. Never commit real secrets.
- **`BETTER_AUTH_API_KEY`** ‚Äî Set only if you enable Better Auth API routes that require it; otherwise leave empty per `.env.example`.

> **Note:** The project expects a single `.env` (see comment in `.env.example`); align with how your host injects env vars.

## Stripe (live mode)

- Switch **`STRIPE_SECRET_KEY`** to a **live** key (`sk_live_...`).
- Create **live** products/prices in Stripe Dashboard and set all four **`STRIPE_PRICE_*`** variables (Pro/Team, monthly/yearly) to live price IDs.
- Register a **live** webhook endpoint pointing to **`https://<your-domain>/api/stripe/webhook`** and set **`STRIPE_WEBHOOK_SECRET`** to the live signing secret (`whsec_...` from the Dashboard or Stripe CLI for testing).

> **Note:** Local testing uses test keys and `stripe listen`; see `docs/STRIPE_TEST_MODE_SETUP.md`. Production must use live keys and a publicly reachable webhook URL.

## Email (Resend)

- **`RESEND_API_KEY`** ‚Äî Production API key.
- **`EMAIL_FROM`** ‚Äî Use a **verified domain** and sender in Resend (format like `Acme <hello@yourdomain.com>`).
- **`EMAIL_REPLY_TO`** ‚Äî Optional; set if you want replies to go somewhere other than the from address.

## OAuth (if enabled)

- **`GOOGLE_CLIENT_ID`** / **`GOOGLE_CLIENT_SECRET`** and/or **GitHub** vars ‚Äî In each provider‚Äôs console, set redirect/callback URLs to your production **`BASE_URL`** (per Better Auth‚Äôs callback paths).
- Review **`ALLOW_EMAIL_ACCOUNT_LINKING`** for your security posture.

## App behavior

- **`NEXT_PUBLIC_ACCOUNT_MODE`** ‚Äî `personal-only` or `organizations-only` as intended for launch.
- Stripe billing toggles (`STRIPE_ENABLE_TRIAL_WITHOUT_CC`, tax, billing address) ‚Äî Match your legal and product requirements.

## Storage

- **Filesystem default** ‚Äî `.storage` under `process.cwd()` is not durable on ephemeral serverless instances. For production scale, set all **`STORAGE_S3_*`** variables so uploads use S3-compatible storage (see [Storage configuration](../storage/configuration.mdx)).
- Ensure the bucket and IAM/R2 tokens match your deployment region and least-privilege policy.

## Database and operations

- Run **`pnpm build`** in CI or locally to confirm a clean production build (`package.json` scripts: `build`, `start`).
- Apply Prisma migrations to production; avoid `db:push` for managed production unless you intentionally accept that workflow.
- Configure backups and monitoring for PostgreSQL.

## Security review

- HTTPS only in production.
- Rotate any keys that ever appeared in logs or tickets.
- Confirm admin and billing routes behave as expected under real sessions.
