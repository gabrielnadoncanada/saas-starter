## Server components and sensitive data

Authenticated areas use **async server layouts and pages** that call session helpers before rendering. The app layout redirects unauthenticated users; dashboard and settings layouts load the user on the server with `getCurrentUser()`; the admin layout uses `requireAdmin()` so only platform admins reach admin routes.

```tsx title="app/(app)/layout.tsx"
import { routes } from "@/shared/constants/routes";
import { getCurrentUser } from "@/shared/lib/auth/get-current-user";

export default async function AppLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const user = await getCurrentUser();

  if (!user) {
    redirect(routes.auth.login);
  }

  return children;
}
```

`getCurrentUser` is implemented with React `cache()` and reads the session via server-only auth code—do not pass raw user records or secrets into client components except minimal, intentional props (for example `UserProvider` sidebar data).

> **Note:** Anything you render in a Client Component or serialize to the client is visible to users. Keep API keys, database URLs, and privileged checks on the server.

## Environment variables

- **Server-only secrets** (never prefixed with `NEXT_PUBLIC_`): for example `AUTH_SECRET`, `RESEND_API_KEY`, `STRIPE_SECRET_KEY`, database URLs, and webhook secrets. These are read in Node (route handlers, server actions, `process.env` in server modules).
- **Public build-time vars** (`NEXT_PUBLIC_*`): embedded in the client bundle. This repo uses them sparingly—for example `NEXT_PUBLIC_ACCOUNT_MODE` drives UI behavior for account mode.

```ts title="shared/config/account.config.ts"
function resolveAccountMode(): AccountMode {
  const raw = process.env.NEXT_PUBLIC_ACCOUNT_MODE;
  if (raw && VALID_MODES.includes(raw as AccountMode)) {
    return raw as AccountMode;
  }
  return "organizations-only";
}
```

Treat `NEXT_PUBLIC_*` as **public**: do not put secrets there.

## Auth checks on the server

- **Layouts and pages**: `getCurrentUser()`, `requireAdmin()`, and organization helpers (for example `ensureActiveOrganization()`) gate access before rendering.
- **Server actions**: authenticated mutations go through `validatedAuthenticatedAction`, which loads the user and rejects unauthenticated calls before running your handler (see [Data validation](./data-validation)).

Admin-only server logic can use `requireAdminAction()` in `features/auth/server/require-admin.ts`, which validates the session with `auth.api.getSession` and throws if unauthorized.

## Input and requests

Validate and authorize **on the server** for every mutation: use Zod with server actions, and enforce organization or billing rules in the same server path that touches the database—do not rely on UI hiding buttons alone.
