This repo is split by ownership first, not by architecture patterns.

## Top-level map

```text
app/            route files only
features/       product behavior grouped by domain
shared/         cross-feature UI and infrastructure
prisma/         schema, migrations, seed
content/docs/   buyer-facing documentation
test/           Vitest coverage
e2e/            Playwright specs
```

## What each folder owns

### `app/`

Only route surface files belong here:

- `page.tsx`
- `layout.tsx`
- `loading.tsx`
- `error.tsx`
- `route.ts`

Routes should stay thin. They fetch data and hand it to `features/`.

Example:

- `app/(app)/dashboard/tasks/page.tsx` fetches tasks
- `features/tasks/components/tasks-page.tsx` renders the feature

### `features/`

This is where you will edit most product behavior.

Important feature folders:

- `features/tasks/` simple CRUD reference feature
- `features/billing/` entitlements, checkout, portal, usage
- `features/users/` platform user management and admin-only user operations
- `features/organizations/` organization membership, invites, switching
- `features/account/` profile, password, account deletion, linked providers
- `features/auth/` auth flows and app-specific auth guards
- `features/admin/` admin shell and overview only
- `features/dashboard/` shared app dashboard navigation and overview cards
- `features/marketing/` landing page, FAQ, pricing, included-items copy

Typical feature shape:

```text
features/tasks/
  actions/
  components/
  server/
  task.schema.ts
  task-display.ts
```

### `shared/`

Shared code used by multiple features.

Main buckets:

- `shared/components/` UI primitives and layout pieces
- `shared/lib/` auth bootstrap, Prisma client, small helpers
- `shared/config/` app-wide configuration such as billing
- `shared/constants/` route constants and terminology
- `shared/hooks/` reusable client hooks
- `shared/types/` shared TS types like `FormActionState`

If a file is only used by one feature, keep it in that feature instead of moving it here.

### `prisma/`

Database ownership:

- `prisma/models/*.prisma` feature-level models
- `prisma/schema.prisma` root Prisma config
- `prisma/migrations/` migration history
- `prisma/seed.ts` demo data

### `content/docs/`

The docs site content. These files are part of the product, not internal notes.

Key sections:

- `content/docs/installation/`
- `content/docs/project-structure/`
- `content/docs/development-guide/`
- `content/docs/billing/`
- `content/docs/testing/`

### `test/`

Vitest coverage grouped by feature:

- `test/tasks/`
- `test/billing/`
- `test/organizations/`

Prefer testing the risky server logic first.

## Where to edit first

If you want to:

- add or edit a product feature: start in `features/<feature>/`
- add a page: start in `app/` and wire it to a feature
- change billing rules or plan limits: start in `shared/config/billing.config.ts`
- change settings flows: start in `app/(app)/settings/*` and the matching `features/account/` or `features/organizations/` file
- change admin screens: start in `app/(app)/admin/*` and the matching domain feature such as `features/users/` or `features/organizations/`
- change landing-page copy: start in `features/marketing/components/`
- change docs: start in `content/docs/`

## Best reference paths

These are the clearest flows to copy when learning the repo:

1. `features/tasks/` for a simple CRUD feature
2. `features/account/` for form-driven settings
3. `features/billing/` for plan and entitlement logic
4. `features/organizations/` for membership and invite flows
