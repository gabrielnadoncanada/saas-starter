## Organization roles vs platform admin

**Workspace roles** (`owner`, `admin`, `member`) are stored on the `Member.role` field (string) and parsed with helpers in `shared/lib/db/enums.ts` (`ORG_ROLES`, `hasOrgRole`, `getPrimaryOrgRole`, etc.).

**Platform administration** uses the user’s global `role` and `isPlatformAdmin()` in `shared/lib/auth/roles.ts` (`PLATFORM_ADMIN_ROLES` includes `"admin"`). That controls access to the admin area in the sidebar—not org membership.

## Session guards

`features/organizations/server/organization-membership.ts` provides:

- **`requireActiveOrganizationMembership()`** — ensures an active org membership; throws `OrganizationMembershipError` if not.
- **`requireActiveOrganizationRole(allowedRoles)`** — ensures membership **and** that the member’s roles intersect `allowedRoles`.

```ts
export async function requireActiveOrganizationRole(allowedRoles: OrgRole[]) {
  const membership = await requireActiveOrganizationMembership();

  if (!hasAnyOrgRole(membership.roles, allowedRoles)) {
    throw new OrganizationMembershipError(
      "You do not have permission to perform this action",
    );
  }

  return membership;
}
```

## What this starter enforces

Server actions for **invites, invitation cancel/resend, member removal, rename, and delete** use **`requireActiveOrganizationRole(["owner"])`** (see `membership.actions.ts`, `rename-organization.actions.ts`, `delete-organization.actions.ts`). In practice, **only owners** can manage members and organization settings in the UI (`getCurrentOrganizationContext()` exposes `isOwner` for pages like members and organization settings).

> **Note:** The invite dialog only offers **member** and **admin** roles; **owner** is assigned to the org creator via Better Auth’s `creatorRole: "owner"` in `auth-config.ts`, not through the invite form.
