---
title: Server actions
description: Patterns for server actions, revalidation, and Zod-validated form actions.
---

Server actions are the mutation boundary for the app. Keep them small.

## Default shape

For product features:

1. put business logic in `features/<name>/server/`
2. put the action in `features/<name>/actions/`
3. mark the file with `"use server"`
4. call `revalidatePath()` after successful writes

Example from the shipped tasks feature:

```ts title="features/tasks/actions/task.actions.ts"
"use server";

import { revalidatePath } from "next/cache";

import {
  createTaskSchema,
  type CreateTaskValues,
} from "@/features/tasks/task.schema";
import { createTask } from "@/features/tasks/server/task-mutations";
import { routes } from "@/shared/constants/routes";
import { validatedAuthenticatedAction } from "@/shared/lib/auth/authenticated-action";
import type { FormActionState } from "@/shared/types/form-action-state";

export type CreateTaskActionState = FormActionState<CreateTaskValues> & {
  task?: Task;
};

export const createTaskAction = validatedAuthenticatedAction(
  createTaskSchema,
  async (data): Promise<CreateTaskActionState> => {
    const task = await createTask(data);
    revalidatePath(routes.app.tasks);
    return { success: "Task created", task };
  },
);
```

## What `validatedAuthenticatedAction` does

`validatedAuthenticatedAction` in `shared/lib/auth/authenticated-action.ts` is the normal choice for signed-in product forms.

It automatically:

- resolves the current user
- validates `FormData` against a Zod schema
- returns `fieldErrors` and sticky `values` on validation failure
- maps billing and organization membership errors into form state

That means most action bodies only need to:

- call the server mutation
- revalidate the page
- return `{ success: "..." }`

## Public actions

For sign-in, sign-up, forgot-password, and reset-password forms, use `validatedPublicAction`.

```ts title="features/auth/actions/public-auth.actions.ts"
export const signInAction = validatedPublicAction(
  signInFormSchema,
  async ({ email, password, callbackUrl }) => {
    await auth.api.signInEmail({
      headers: await headers(),
      body: {
        email: email.trim().toLowerCase(),
        password,
        callbackURL: callbackUrl,
      },
    });

    return { redirectTo: callbackUrl ?? routes.auth.postSignIn };
  },
);
```

## Owner-only actions

When only organization owners should be allowed to mutate something, use `validatedOrganizationOwnerAction`. It composes the normal auth + validation pipeline with an ownership check for the active organization.

Use this for flows like:

- deleting an organization
- opening billing portal access that must stay owner-only
- ownership-restricted settings

## Admin actions

Admin actions are different because they are not always plain form posts. For admin mutations and programmatic admin UI calls, start with `requireAdminAction()` and then call the domain function directly.

```ts
await requireAdminAction();
return listAdminUsers(query);
```

## Revalidation rules

After a successful mutation:

- revalidate the route the user is currently on
- revalidate any sibling routes that show the same stale data

Do not revalidate inside low-level `server/` functions. Keep that decision in the action layer where the route context is obvious.
