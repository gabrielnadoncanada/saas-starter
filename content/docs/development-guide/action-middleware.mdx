---
title: Action middleware and guards
description: Guards for admin server actions and authenticated form actions.
---

This starter uses a few small guard helpers instead of a custom action framework.

## `requireAdminAction`

Use `requireAdminAction()` at the top of admin mutations that should throw when the caller is not a platform admin.

Typical use:

```ts
await requireAdminAction();
return listAdminUsers(query);
```

This is the right fit for programmatic admin actions and non-form admin calls.

## `requireAdmin`

Use `requireAdmin()` in Server Components and layouts. It redirects instead of throwing, which makes it better for page-level protection.

Use it in:

- `app/(app)/admin/layout.tsx`
- admin page loaders
- admin-only dashboards

## `validatedAuthenticatedAction`

Use `validatedAuthenticatedAction()` for normal signed-in product forms.

It handles:

- session lookup
- Zod validation
- sticky form values
- field-level errors
- billing and organization membership error mapping

That keeps the action body simple and lets the form consume one predictable `FormActionState`.

## `validatedOrganizationOwnerAction`

Use `validatedOrganizationOwnerAction()` when the caller must be the owner of the active organization. It builds on `validatedAuthenticatedAction()` and adds the ownership requirement.

This is the clean default for:

- deleting an organization
- owner-only billing flows
- restricted organization settings

## Domain guards still belong in `server/`

Action helpers are not a substitute for domain checks.

The actual business boundary should still live in `features/<name>/server/`. For example, `features/tasks/server/task-mutations.ts` checks organization membership and billing entitlements before writing any data.

Use this rule:

- auth + form validation at the action boundary
- business rules at the domain boundary
