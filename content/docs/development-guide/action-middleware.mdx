## Admin actions: `requireAdminAction`

Server actions that perform platform admin work must assert an admin session before doing anything. The shared helper lives in `features/auth/server/require-admin.ts`:

```ts title="features/auth/server/require-admin.ts"
export async function requireAdminAction(): Promise<string> {
  const session = await auth.api.getSession({
    headers: await headers(),
  });

  if (!session?.user?.id) {
    throw new Error("Unauthorized");
  }

  if (!isPlatformAdmin(session.user.role)) {
    throw new Error("Forbidden: admin role required");
  }

  return session.user.id;
}
```

Return value is the **admin user id**, used when an action must prevent self-targeted operations (for example banning or deleting your own account).

Use this at the **start** of every admin action:

```ts title="features/admin/actions/admin-users.actions.ts"
export async function listAdminUsersAction(query: ListAdminUsersQuery) {
  await requireAdminAction();
  return listAdminUsers(query);
}
```

## Page-level admin guard: `requireAdmin`

For Server Components and layouts, use `requireAdmin()` instead. It redirects to sign-in or the main dashboard rather than throwing:

```ts title="features/auth/server/require-admin.ts"
export async function requireAdmin() {
  const user = await getCurrentUser();

  if (!user) {
    redirect(routes.auth.login);
  }

  if (!isPlatformAdmin(user.role)) {
    redirect(routes.app.dashboard);
  }

  return user;
}
```

## Authenticated form pipeline: `validatedAuthenticatedAction`

For normal (signed-in) app users submitting forms, use `validatedAuthenticatedAction` from `shared/lib/auth/authenticated-action.ts`. It composes:

- Session check via `getCurrentUser()`.
- Zod `safeParse` on `Object.fromEntries(formData)`.
- A typed handler that receives parsed data, raw `FormData`, and the user object.

```ts title="shared/lib/auth/authenticated-action.ts"
export function validatedAuthenticatedAction<
  S extends z.ZodTypeAny,
  TExtraState extends object = {},
>(schema: S, action: ValidatedAuthenticatedActionHandler<S, TExtraState>) {
  type Values = z.infer<S>;
  type State = FormActionState<Values> & TExtraState;

  return async (_prevState: State, formData: FormData): Promise<State> => {
    const user = await getCurrentUser();

    if (!user) {
      return {
        error: NOT_AUTHENTICATED,
      } as State;
    }

    const rawValues = Object.fromEntries(formData) as Record<string, unknown>;
    const parsed = schema.safeParse(rawValues);

    if (!parsed.success) {
      const flat = parsed.error.flatten();

      return {
        error: FORM_VALIDATION_FAILED,
        values: rawValues as Partial<Values>,
        fieldErrors: flat.fieldErrors as State["fieldErrors"],
      } as State;
    }

    return action(parsed.data, formData, user);
  };
}
```

Pair this with `useActionState` in a client component and a `<form action={formAction}>` for progressive enhancement.

## Domain guards inside server modules

Additional checks live next to data access. For example, `features/tasks/server/task-mutations.ts` uses `requireActiveOrganizationMembership()` and billing helpers so mutations only apply within the active org and plan limits. That is not a generic “middleware” but the same idea: **guard at the boundary you own**.

> **Note:** Throwing in `requireAdminAction` surfaces as errors to the caller; admin UI wraps actions in transitions and toasts. Design user-facing messaging in those call sites.
