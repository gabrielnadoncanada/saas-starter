title: Forms
description: Standard form pattern with Zod, server actions, and shared field state.
---

Use one form model across the app:

- schema lives in the feature
- server action owns auth + validation
- client form uses `useActionState`
- action returns `FormActionState<TValues>`
- fields read `state.fieldErrors` and `state.values`

## The default pattern

The shipped task form is the baseline CRUD example:

```tsx title="features/tasks/components/task-form-sheet.tsx"
const isUpdate = Boolean(task);
const action = isUpdate ? updateTaskAction : createTaskAction;

const [state, formAction, isPending] = useActionState<TaskFormState, FormData>(
  action,
  {},
);
```

Then render shadcn field primitives directly:

```tsx title="features/tasks/components/task-form-sheet.tsx"
const titleError = state.fieldErrors?.title?.[0];
const titleValue = String(state.values?.title ?? task?.title ?? "");

<Field data-invalid={Boolean(titleError)}>
  <FieldLabel htmlFor="task-title">Title</FieldLabel>
  <Input
    id="task-title"
    name="title"
    defaultValue={titleValue}
    aria-invalid={Boolean(titleError)}
    required
  />
  <FieldError>{titleError}</FieldError>
</Field>;
```

That is enough for most starter forms. Do not add React Hook Form, custom field registries, or generic form wrappers unless the form is genuinely more complex than the shipped dialogs.

## Shared state contract

Every app form returns `FormActionState<TValues>` from `shared/types/form-action-state.ts`.

That gives you:

- `success` for success toasts
- `error` for top-level failures
- `errorCode` for billing upgrade prompts
- `values` for sticky inputs after validation failure
- `fieldErrors` for inline errors

## Authenticated vs public actions

- Use `validatedAuthenticatedAction` for signed-in product forms such as tasks, settings, and billing-adjacent flows.
- Use `validatedPublicAction` for auth pages such as sign-in, sign-up, forgot password, and reset password.

Authenticated example:

```ts title="features/tasks/actions/task.actions.ts"
export const createTaskAction = validatedAuthenticatedAction(
  createTaskSchema,
  async (data): Promise<CreateTaskActionState> => {
    const task = await createTask(data);
    revalidatePath(routes.app.tasks);
    return { success: "Task created", task };
  },
);
```

Public example:

```ts title="features/auth/actions/public-auth.actions.ts"
export const signInAction = validatedPublicAction(
  signInFormSchema,
  async ({ email, password, callbackUrl }) => {
    await auth.api.signInEmail({
      headers: await headers(),
      body: {
        email: email.trim().toLowerCase(),
        password,
        callbackURL: callbackUrl,
      },
    });

    return { redirectTo: callbackUrl ?? routes.auth.postSignIn };
  },
);
```

## Field helpers

Many forms read `state.fieldErrors?.fieldName?.[0]` directly, like `TaskFormSheet`.

When a form has several plain inputs, `getFieldState()` in `shared/lib/get-field-state.ts` can reduce repetition:

```ts
const emailField = getFieldState(state, "email");
const roleField = getFieldState(state, "role", "member");
```

Use the helper when it makes the component shorter. Do not build a mini form framework around it.

## Keep business logic out of the client

The client form should only:

- render fields
- submit to the action
- react to `success` and `error`

Database writes, entitlement checks, and organization guards belong in `features/<name>/server/`.
