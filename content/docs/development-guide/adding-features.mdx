This guide walks through adding a new feature end-to-end. We'll build a fictional **invoices** feature so you can see every layer touched. By the end you'll know where each file goes and which shared helpers to call.

The starter already ships a complete **tasks** feature that demonstrates all of these patterns — when in doubt, read `features/tasks/` alongside this guide.

## Feature folder layout

Every feature lives under `features/<name>/` with the same subfolders:

```
features/invoices/
  actions/             → "use server" server actions called from the client
  components/          → React UI (client + server components)
  server/              → server-only helpers (Prisma, guards, mutations)
  invoices.schema.ts   → Zod schemas + derived types (flat file is fine)
  types.ts             → optional: shared TypeScript types
```

Dependency direction is strict: **`app/` → `features/` → `shared/lib/`**. Features never import from `app/`, and they rarely import from each other — promote shared code to `shared/lib/` when two features need it.

## Step 1 — Model the data

Add the Prisma model to the appropriate file in `prisma/models/`. Create a new file if your feature doesn't fit an existing group.

```prisma title="prisma/models/invoices.prisma"
model Invoice {
  id             String   @id @default(cuid())
  organizationId String
  number         String
  amountCents    Int
  status         InvoiceStatus @default(DRAFT)
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  organization Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)

  @@unique([organizationId, number])
  @@index([organizationId, createdAt])
}

enum InvoiceStatus {
  DRAFT
  SENT
  PAID
  VOID
}
```

Run the migration and regenerate the client:

```bash
pnpm db:migrate
pnpm db:generate
```

Every tenant-scoped model **must** carry `organizationId` and a compound index starting with it. All queries filter on `organizationId`, and the starter enforces this via the guards in Step 3.

## Step 2 — Write the Zod schema

Schemas live next to the feature. Keep form schemas and derived types in the same file so actions and components can import from one place.

```ts title="features/invoices/invoices.schema.ts"
import { z } from "zod";

import { InvoiceStatus } from "@/shared/lib/db/enums";

export const createInvoiceSchema = z.object({
  number: z.string().trim().min(1, { message: "Number is required" }).max(64),
  amountCents: z.coerce.number().int().positive(),
  status: z.nativeEnum(InvoiceStatus).default(InvoiceStatus.DRAFT),
});

export const updateInvoiceSchema = createInvoiceSchema.extend({
  invoiceId: z.string().cuid(),
});

export type CreateInvoiceValues = z.infer<typeof createInvoiceSchema>;
export type UpdateInvoiceValues = z.infer<typeof updateInvoiceSchema>;
```

Schemas do double duty: they validate server-action input and produce the `TValues` type that drives `FormActionState<TValues>` on the client.

## Step 3 — Server mutations and queries

Server helpers go in `features/invoices/server/`. Add `import "server-only"` at the top so Next.js fails the build if a client component imports it by mistake.

Use `getCurrentEntitlements()` to get both the active organization **and** the billing plan in one call. The returned `entitlements` object is what you pass to `assertCapability`, `assertLimit`, and `consumeMonthlyUsage`.

```ts title="features/invoices/server/invoice-mutations.ts"
import "server-only";

import { getCurrentEntitlements } from "@/features/billing/server/organization-entitlements";
import { prisma } from "@/shared/lib/db/prisma";

import type { CreateInvoiceValues } from "@/features/invoices/invoices.schema";

export async function createInvoice(data: CreateInvoiceValues) {
  const entitlements = await getCurrentEntitlements();

  return prisma.invoice.create({
    data: {
      ...data,
      organizationId: entitlements.organizationId,
    },
  });
}

export async function listInvoices() {
  const entitlements = await getCurrentEntitlements();

  return prisma.invoice.findMany({
    where: { organizationId: entitlements.organizationId },
    orderBy: { createdAt: "desc" },
  });
}
```

### Gating a feature by plan

If the feature requires a paid plan or has a usage limit, guard it inside the mutation — never rely on the client alone. The guards throw `UpgradeRequiredError` / `LimitReachedError`, which the action wrapper translates into a friendly error state (Step 4).

```ts
import {
  assertCapability,
  assertLimit,
} from "@/features/billing/server/billing-guards";

export async function createInvoice(data: CreateInvoiceValues) {
  const entitlements = await getCurrentEntitlements();

  assertCapability(entitlements, "invoices.create");
  await assertLimit(entitlements, "invoices.perMonth");

  return prisma.invoice.create({ /* ... */ });
}
```

Capabilities and limits are declared in `features/billing/plans.ts` — add your new keys there and they'll flow through to every plan.

## Step 4 — Server action

Actions sit in `features/invoices/actions/` and are the only server entry point the client talks to. Wrap every action in `validatedAuthenticatedAction` — it runs auth, parses the Zod schema, and returns the discriminated `FormActionState` the forms expect.

Wrap any body that can raise billing errors in `withBillingErrors` so the client receives `errorCode: "UPGRADE_REQUIRED" | "LIMIT_REACHED"` instead of a 500.

```ts title="features/invoices/actions/invoice.actions.ts"
"use server";

import type { Invoice } from "@prisma/client";
import { revalidatePath } from "next/cache";

import {
  createInvoiceSchema,
  type CreateInvoiceValues,
} from "@/features/invoices/invoices.schema";
import { createInvoice } from "@/features/invoices/server/invoice-mutations";
import { routes } from "@/shared/constants/routes";
import {
  validatedAuthenticatedAction,
  withBillingErrors,
} from "@/shared/lib/auth/authenticated-action";
import type { FormActionState } from "@/shared/types/form-action-state";

export type CreateInvoiceActionState = FormActionState<CreateInvoiceValues> & {
  invoice?: Invoice;
};

export const createInvoiceAction = validatedAuthenticatedAction<
  typeof createInvoiceSchema,
  { invoice?: Invoice }
>(createInvoiceSchema, async (data) => {
  return withBillingErrors(async () => {
    const invoice = await createInvoice(data);

    revalidatePath(routes.app.invoices);

    return { success: "Invoice created", invoice };
  });
});
```

## Step 5 — Build the form

Forms use `useActionState` bound to the server action. The `FormActionState<TValues>` shape is what `FormField` reads to render per-field errors without extra glue code.

```tsx title="features/invoices/components/create-invoice-form.tsx"
"use client";

import { useActionState } from "react";

import {
  createInvoiceAction,
  type CreateInvoiceActionState,
} from "@/features/invoices/actions/invoice.actions";
import { FormField } from "@/shared/components/forms/form-field";
import { FormStatus } from "@/shared/components/forms/form-status";
import { SubmitButton } from "@/shared/components/forms/submit-button";

const initialState: CreateInvoiceActionState = {};

export function CreateInvoiceForm() {
  const [state, formAction] = useActionState(createInvoiceAction, initialState);

  return (
    <form action={formAction} className="space-y-4">
      <FormField label="Number" name="number" state={state} />
      <FormField label="Amount (cents)" name="amountCents" state={state} type="number" />
      <FormStatus state={state} />
      <SubmitButton>Create invoice</SubmitButton>
    </form>
  );
}
```

`FormField` is wired to read `state.fieldErrors[name]` automatically. `FormStatus` renders `state.success`, `state.error`, and upgrade prompts for `errorCode: "UPGRADE_REQUIRED"`.

## Step 6 — Wire up the route

Routes stay thin: auth/org resolution, data fetch, compose feature components. Put route files under `app/(app)/invoices/`.

```tsx title="app/(app)/invoices/page.tsx"
import { CreateInvoiceForm } from "@/features/invoices/components/create-invoice-form";
import { InvoicesTable } from "@/features/invoices/components/invoices-table";
import { listInvoices } from "@/features/invoices/server/invoice-mutations";
import { Page, PageHeader } from "@/shared/components/layout/page-layout";

export default async function InvoicesPage() {
  const invoices = await listInvoices();

  return (
    <Page>
      <PageHeader title="Invoices" description="Manage your billing documents." />
      <CreateInvoiceForm />
      <InvoicesTable invoices={invoices} />
    </Page>
  );
}
```

Add the route to `shared/constants/routes.ts` so every link is type-checked:

```ts
export const routes = {
  app: {
    // ...
    invoices: "/invoices",
  },
};
```

## Step 7 — Navigation and access

Add the entry to `features/dashboard/config/dashboard-navigation.ts` so it appears in the sidebar. If the feature is plan-gated, guard the route itself with `hasCapability(entitlements, "invoices.create")` and render the `<UpgradeCard />` when the user lacks access — the assistant page at `app/(app)/assistant/page.tsx` is a concise reference for this pattern.

## Step 8 — Tests

Feature tests live under `test/<feature>/`. Unit-test mutations by calling them against a real database (the test harness spins up a scratch schema) and integration-test actions by invoking them through the Next.js action boundary.

```ts title="test/invoices/invoice-mutations.test.ts"
import { describe, expect, it } from "vitest";

import { createInvoice } from "@/features/invoices/server/invoice-mutations";

describe("createInvoice", () => {
  it("persists the invoice with the active organization id", async () => {
    const invoice = await createInvoice({
      number: "INV-001",
      amountCents: 9900,
      status: "DRAFT",
    });

    expect(invoice.organizationId).toBeTruthy();
  });
});
```

Run the suite with `pnpm test` or `pnpm test:watch`.

## Checklist

A new feature is "done" when all of these are true:

- [ ] Prisma model carries `organizationId` and a compound index
- [ ] Zod schema in `features/<name>/<name>.schema.ts` with exported `TValues` types
- [ ] Server mutations `import "server-only"` and call `getCurrentEntitlements()`
- [ ] Plan-gated paths call `assertCapability` / `assertLimit` / `consumeMonthlyUsage`
- [ ] Server actions wrap bodies in `withBillingErrors` where billing errors can fire
- [ ] Route file under `app/(app)/...` is thin and calls into `features/`
- [ ] Route added to `shared/constants/routes.ts`
- [ ] Navigation entry added (if user-facing)
- [ ] Tests under `test/<name>/`
