---
title: Adding a feature
description: Add a new feature using the same boring CRUD path the starter ships in the tasks module.
---

Start with the same default path used by the shipped `tasks` feature:

1. one Prisma model
2. one schema file
3. one `server/` file for queries and mutations
4. one `actions/` file
5. one main page component
6. one form component
7. one thin route under `app/`

If your first version needs sorting, filters, URL state, or bulk actions, stop and ask whether the feature is really advanced enough to earn that complexity. For most starter features, it is not.

## Folder shape

Keep the feature local and obvious:

```text
features/invoices/
  actions/
    invoice.actions.ts
  components/
    invoice-form-sheet.tsx
    invoices-page.tsx
  server/
    invoice-mutations.ts
  invoice.schema.ts
```

Routes stay thin and live under `app/` only:

```text
app/(app)/dashboard/invoices/page.tsx
```

## Step 1: model the data

Add the Prisma model in `prisma/models/`. Tenant-scoped models should carry `organizationId` and index it.

```prisma title="prisma/models/invoices.prisma"
model Invoice {
  id             String   @id @default(cuid())
  organizationId String
  number         String
  amountCents    Int
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  organization Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)

  @@unique([organizationId, number])
  @@index([organizationId, createdAt])
}
```

Then run:

```bash
pnpm db:migrate
pnpm db:generate
```

## Step 2: add the feature schema

Keep Zod schemas next to the feature. The same file powers validation and form types.

```ts title="features/invoices/invoice.schema.ts"
import { z } from "zod";

export const createInvoiceSchema = z.object({
  number: z.string().trim().min(1, { message: "Number is required" }).max(64),
  amountCents: z.coerce.number().int().positive(),
});

export const updateInvoiceSchema = createInvoiceSchema.extend({
  invoiceId: z.string().cuid(),
});

export type CreateInvoiceValues = z.infer<typeof createInvoiceSchema>;
export type UpdateInvoiceValues = z.infer<typeof updateInvoiceSchema>;
```

## Step 3: keep server logic in one obvious file

Put queries and mutations in `server/` and mark the file `server-only`.

```ts title="features/invoices/server/invoice-mutations.ts"
import "server-only";

import { assertCapability } from "@/features/billing/entitlements";
import { getCurrentEntitlements } from "@/features/billing/server/organization-entitlements";
import { db } from "@/shared/lib/db/prisma";

import type { CreateInvoiceValues } from "@/features/invoices/invoice.schema";

export async function createInvoice(input: CreateInvoiceValues) {
  const entitlements = await getCurrentEntitlements();

  if (!entitlements) {
    throw new Error("Organization not found");
  }

  assertCapability(entitlements, "invoice.create");

  return db.invoice.create({
    data: {
      organizationId: entitlements.organizationId,
      number: input.number,
      amountCents: input.amountCents,
    },
  });
}

export async function listInvoices() {
  const entitlements = await getCurrentEntitlements();

  return db.invoice.findMany({
    where: { organizationId: entitlements.organizationId },
    orderBy: { createdAt: "desc" },
  });
}
```

Billing and organization guards belong here, not in the client.

## Step 4: wrap mutations in one server action file

Client forms should talk to `actions/`, not directly to `server/`.

```ts title="features/invoices/actions/invoice.actions.ts"
"use server";

import { revalidatePath } from "next/cache";

import {
  createInvoiceSchema,
  type CreateInvoiceValues,
} from "@/features/invoices/invoice.schema";
import { createInvoice } from "@/features/invoices/server/invoice-mutations";
import { routes } from "@/shared/constants/routes";
import { validatedAuthenticatedAction } from "@/shared/lib/auth/authenticated-action";
import type { FormActionState } from "@/shared/types/form-action-state";

export type CreateInvoiceActionState = FormActionState<CreateInvoiceValues>;

export const createInvoiceAction = validatedAuthenticatedAction(
  createInvoiceSchema,
  async (data): Promise<CreateInvoiceActionState> => {
    await createInvoice(data);
    revalidatePath(routes.app.invoices);
    return { success: "Invoice created" };
  },
);
```

`validatedAuthenticatedAction` already:

- checks the session
- validates `FormData` with Zod
- returns `fieldErrors` and sticky `values`
- maps billing and organization errors to form state automatically

You do not need a second billing wrapper around the action body.

## Step 5: build a small form component

Use `useActionState`, shadcn field primitives, and inline field errors.

```tsx title="features/invoices/components/invoice-form-sheet.tsx"
"use client";

import { useActionState } from "react";

import {
  createInvoiceAction,
  type CreateInvoiceActionState,
} from "@/features/invoices/actions/invoice.actions";
import { Button } from "@/shared/components/ui/button";
import {
  Field,
  FieldError,
  FieldGroup,
  FieldLabel,
} from "@/shared/components/ui/field";
import { Input } from "@/shared/components/ui/input";

const initialState: CreateInvoiceActionState = {};

export function InvoiceForm() {
  const [state, formAction, isPending] = useActionState(
    createInvoiceAction,
    initialState,
  );

  const numberError = state.fieldErrors?.number?.[0];
  const amountError = state.fieldErrors?.amountCents?.[0];

  return (
    <form action={formAction} className="space-y-6">
      <FieldGroup className="gap-4">
        <Field data-invalid={Boolean(numberError)}>
          <FieldLabel htmlFor="invoice-number">Number</FieldLabel>
          <Input
            id="invoice-number"
            name="number"
            defaultValue={String(state.values?.number ?? "")}
            aria-invalid={Boolean(numberError)}
            required
          />
          <FieldError>{numberError}</FieldError>
        </Field>

        <Field data-invalid={Boolean(amountError)}>
          <FieldLabel htmlFor="invoice-amount">Amount (cents)</FieldLabel>
          <Input
            id="invoice-amount"
            name="amountCents"
            type="number"
            defaultValue={String(state.values?.amountCents ?? "")}
            aria-invalid={Boolean(amountError)}
            required
          />
          <FieldError>{amountError}</FieldError>
        </Field>
      </FieldGroup>

      <Button type="submit" disabled={isPending}>
        {isPending ? "Saving..." : "Create invoice"}
      </Button>
    </form>
  );
}
```

If you need a bigger example, read `features/tasks/components/task-form-sheet.tsx`.

## Step 6: keep the route thin

The route should fetch data and hand it to the feature page. Nothing more.

```tsx title="app/(app)/dashboard/invoices/page.tsx"
import { InvoicesPage } from "@/features/invoices/components/invoices-page";
import { listInvoices } from "@/features/invoices/server/invoice-mutations";

export default async function DashboardInvoicesPage() {
  const invoices = await listInvoices();

  return <InvoicesPage invoices={invoices} />;
}
```

Add the route constant in `shared/constants/routes.ts` and sidebar entry in `features/dashboard/config/dashboard-navigation.ts` if the feature should appear in navigation.

## Step 7: keep the first list boring

For the first version, prefer a direct table or list inside `features/invoices/components/invoices-page.tsx`.

Good default:

- fetch all rows with one `listInvoices()` query
- render a simple table or empty state
- open create/edit/delete dialogs from that page component

Avoid by default:

- URL-state helpers
- generic table frameworks
- bulk actions
- server-side pagination
- feature-local filter systems

Use the admin tables as the advanced example later, not as the starting point.

## Step 8: add tests where the logic lives

`test/<feature>/` should focus on the risky behavior: mutations, plan gating, and form behavior.

```ts title="test/invoices/create-invoice.test.ts"
import { describe, expect, it } from "vitest";

import { createInvoice } from "@/features/invoices/server/invoice-mutations";

describe("createInvoice", () => {
  it("persists the invoice in the active organization", async () => {
    const invoice = await createInvoice({
      number: "INV-001",
      amountCents: 9900,
    });

    expect(invoice.organizationId).toBeTruthy();
  });
});
```

## Checklist

- [ ] Prisma model exists and is scoped correctly
- [ ] Schema lives in `features/<name>/<name>.schema.ts`
- [ ] `server/` owns queries and mutations
- [ ] `actions/` owns the server action boundary
- [ ] The first page version is a direct list or table, not a framework
- [ ] Route file in `app/` is thin
- [ ] Route constant and navigation entry are added when needed
- [ ] Tests cover the risky server logic
