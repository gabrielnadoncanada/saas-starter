## What “advanced” means here

The starter does **not** configure Stripe graduated/tiered price tiers in code. Instead, it models:

- **Per-interval schedules** (`month` / `year`) with one or more **line items**.
- Each line item has a **`kind`**: `"flat"` (quantity 1 in checkout) or `"seat"` (quantity derived from seat selection).
- Optional **`includedQuantity`** on seat lines reduces billed seats before the minimum of 1 applies.

```ts title="shared/config/billing.config.ts"
export type BillingRecurringLineItem = {
  id: string;
  kind: BillingLineItemKind;
  price: BillingPrice;
  includedQuantity?: number;
  unitLabel?: string;
};

export type BillingPlanSchedule = {
  lineItems: readonly BillingRecurringLineItem[];
};
```

## Building checkout quantities

`buildRecurringSelectionItems` maps each line item to a Stripe line: flat lines use quantity `1`; seat lines use:

`max(1, seatQuantity - (includedQuantity ?? 0))`.

```ts title="features/billing/catalog.ts"
  return schedule.lineItems.map((lineItem) => ({
    componentKey: lineItem.id,
    itemKey: plan.id,
    itemType: "plan" as const,
    priceId: lineItem.price.priceId,
    quantity:
      lineItem.kind === "seat"
        ? Math.max(1, params.seatQuantity - (lineItem.includedQuantity ?? 0))
        : 1,
  }));
```

## Catalog index for webhooks

`catalog.ts` indexes all configured recurring prices so webhook sync can resolve Stripe price ids back to plan metadata via `findCatalogRecurringPriceByPriceId`.

## One-time products

`OneTimeProductDefinition` supports optional `price` (Stripe price id + unit amount for display) and `limitEffect` to bump plan limits (for example extra storage MB) after a completed payment purchase.

> **Note:** To model multiple recurring components per plan (e.g. base + add-on), add multiple entries to `lineItems`. The Team plan in the default config uses a single `seat` line item.
