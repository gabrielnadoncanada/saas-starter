---
title: Session handling
description: How the app reads the Better Auth session and resolves the current user on the server.
---

Session helpers live in **`shared/lib/auth/`** (not under `features/auth/server/`, which holds admin/onboarding/password helpers). They are thin wrappers around **Better Auth**’s `auth.api.getSession` and React **`cache`** for per-request deduplication.

## `getAuthSession` (`shared/lib/auth/get-session.ts`)

```ts
export const getAuthSession = cache(async (): Promise<AuthSession | null> => {
  const session = await auth.api.getSession({
    headers: await headers(),
  });

  return session as AuthSession | null;
});
```

- Uses Next.js **`headers()`** so the session matches the incoming request.
- **`AuthSession`** extends the raw Better Auth session type so **`activeOrganizationId`** is typed on `session` when present.
- Wrapped in React **`cache`** so multiple calls in the same request share one result.

## `getCurrentUser` (`shared/lib/auth/get-current-user.ts`)

```ts
export const getCurrentUser = cache(async (): Promise<CurrentUser | null> => {
  const session = await getAuthSession();

  if (!session?.user?.id) {
    return null;
  }

  return {
    ...session.user,
    role: session.user.role ?? null,
    twoFactorEnabled: session.user.twoFactorEnabled ?? false,
  };
});
```

- Returns **`null`** when there is no authenticated user.
- **`CurrentUser`** is the session `user` plus normalized **`role`** and **`twoFactorEnabled`** for UI and guards.
- **`toDisplayUser`** maps a `CurrentUser` to a smaller shape for navigation.

## Usage patterns

- **Server Components & server actions:** Call `getAuthSession()` when you need organization context or the raw session; call `getCurrentUser()` when you only need the user row fields.
- **Authorization helpers:** For example `features/auth/server/require-admin.ts` imports `getCurrentUser` to enforce admin access.

> **Note:** These functions are **async** and must run on the server. Do not import them into client components; use Better Auth’s client (`authClient`) in the browser instead.

> **Warning:** `getAuthSession` depends on request headers. Use it inside the Next.js request context (RSC, Server Actions, Route Handlers), not in arbitrary background jobs unless you pass a compatible request context.
